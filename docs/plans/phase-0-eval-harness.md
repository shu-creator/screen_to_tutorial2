# Phase 0: 評価基盤 + ベースライン測定

ステータス: 実装済み（2026-06-12）。ただし以下は**実データ待ち**:
- 現行パイプラインの実ベースライン測定（LLM APIキー未設定のため。`eval/baseline.json` は pending マーカー）
- 早期実験（bbox+前フレーム注入の効果測定）— 同上
- 実録画ケースの追加（合成3ケースで開始。追加手順は `eval/README.md`）

規模: M
依存: なし

## 目的

「良いマニュアルが出ているか」を数値で測れるようにする。以降の全フェーズは、この物差しでベースライン比の改善を確認してからマージする。

## 現状の問題

- テスト109件はテキスト整形ユーティリティ等の単体テストのみで、生成品質を測るものがゼロ（要件定義 C9）
- 改善施策の効果を主観でしか判断できず、`slideText.ts` のような対症療法が積み上がる温床になっている

## スコープ

### 1. 評価データセット
- 録画 5〜10本 + 人手正解。最初の3本は「ステップ境界（ms単位の区間リスト）+ステップタイトル」だけの簡易正解で開始し、徐々に拡充する
- カバすべきバリエーション:
  - ナレーション付き / 無音（前提: 入力は混在）
  - フォーム入力中心（タイピング合体の検証用）
  - モーダル/ドロップダウン遷移を含むもの（フェード遷移の検証用）
  - 短尺（〜2分）と中尺（〜10分）
- 置き場所: `eval/dataset/<case-id>/`（`ground_truth.json` + `meta.json` はコミット、動画実体は `.gitignore` にエントリを追加した上で `eval/dataset/<case-id>/video.mp4` に置き、`meta.json` にSHA-256と入手方法を記録する）

### 2. 正解データスキーマ（`ground_truth.json`）
```jsonc
{
  "version": 1,
  "case_id": "form-entry-narrated-01",
  "steps": [
    {
      "t_start": 0,        // ms
      "t_end": 4200,       // ms
      "title": "ログイン画面を開く",
      "ui_labels": ["ログイン", "メールアドレス"],  // このステップで言及されるべきUIラベル（OCR照合の正解）
      "non_step": false     // スクロール・待機などマニュアルに載せるべきでない区間は true
    }
  ]
}
```

### 3. メトリクス実装（`eval/metrics.ts`）
- **G1 ステップ分割F1**: 生成ステップと正解ステップを区間IoU（閾値0.5）でマッチングし、Precision/Recall/F1を算出
- **G2 UIラベル正確性**: 生成文（title/operation/instruction）から「」『』等で引用されたUIラベルを抽出し、正解 `ui_labels` ∪ OCR実測テキストに含まれる率。**分母0の規約**: 現行プロンプトは括弧引用を指示していないため、ベースラインでは引用0件のステップが多発しうる。引用0件のステップはG2の分母から除外し、代わりに「引用なしステップ率」を必ず併記する（引用ゼロでG2が見かけ100%になる退化を防ぐ）
- **G3 非ステップ混入率**: 正解で `non_step: true` の区間にマッチした生成ステップ数 / 生成ステップ総数
- G4（人手修正コスト）は自動化せず、評価実行時に記録欄を出力するに留める

### 4. 評価ランナー（`pnpm eval`）
- 入力: 評価セットのケースID（省略時は全件）
- 動作: CLIパイプライン（既存 `pnpm pipeline:generate` を流用）で steps.json を生成し、正解と突き合わせてメトリクスを計算
- 出力: `eval/results/<timestamp>.json` + コンソールのサマリーテーブル + ベースライン（`eval/baseline.json`）との差分表示
- LLM呼び出しは既存のパイプラインキャッシュが効くため、2回目以降はAPIコストなしで再実行できる

### 5. ベースライン測定と早期改善実験
1. 現行パイプラインで全ケースを実行し、`eval/baseline.json` として固定する
2. **早期実験**: 現行 `analyzeFrame()` に (a) 前フレーム画像、(b) `detectChangedRegion` の差分bbox、(c) 直前ステップの生成結果、を注入した実験ブランチを作り、G1〜G3の変化を測る
   - この実験の結果は Phase 2 のプロンプト設計（何の証拠がどれだけ効くか）の根拠になる
   - 効果が大きければ、Phase 1-2 完成を待たずに本線へ取り込んでよい（低コスト改修のため）

## やらないこと

- 評価セットの大規模化（10本超）
- 人手評価のクラウドソーシング
- LLM-as-judge による文章品質スコア（将来検討。まず決定的メトリクスを固める）

## リスク

| リスク | 対応 |
|--------|------|
| 正解データ作成が重い | 簡易正解（境界+タイトルのみ）から開始。G2用の `ui_labels` は後追いで付与 |
| 録画素材の権利/機密 | 自作のデモ録画（OSSアプリ操作等）のみ使用する |
| 区間IoUマッチングの実装バグで誤った結論を出す | メトリクス自体に単体テストを書く（受け入れ基準に含む） |
