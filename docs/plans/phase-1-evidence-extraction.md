# Phase 1: 証拠抽出パイプライン刷新（evidence.json）

ステータス: 未着手
規模: L（PR 2〜3本に分割推奨）
依存: Phase 0（評価基盤でセグメント境界品質を測れること）

## 目的

「操作セグメント」を正しく切り出し、各セグメントに執筆フェーズが必要とする証拠（before/after画像・差分bbox・OCR・発話）を決定的に紐付ける。LLMを使わない安価で再現的な前処理に徹する。

## 現状の問題（置き換え対象）

- `videoProcessor.ts` `extractFramesWithFFmpeg()`: ffmpeg `select=gt(scene,t)` は映像カット検出用で、タイピング（小差分の連続）、モーダルのフェードイン（遷移途中フレームを掴む）、ページ内アニメーション（誤検出）に弱い（C4）
- `extractBestFrame()`: 変化点の+0.3/0.6/1.0秒後からJPEGファイルサイズ最大を選ぶ。鮮明さの代理として粗く、1秒後には次の操作が始まっている場合に誤った画面状態を代表に選ぶ（C5）
- dHash重複除去（`frameAnalysis.ts`）はシーン検出の**後段**の救済にすぎず、検出漏れは救えない
- 差分bbox（`detectChangedRegion`）は計算されるがスライド装飾にしか使われない（C3）
- ASRスニペットがフレーム時刻**以降**のみで、操作に先行する発話を取り逃す（C6）

## 設計

### 処理フロー

```
動画 ─► (1) 一様サンプリング 2〜4fps（ffmpeg、ディスクへ連番JPEG）
      ─► (2) dHashタイムライン構築（全サンプルフレーム）
      ─► (3) 変化点検出: ハミング距離が閾値を超える区間を「遷移」、下回る連続区間を「安定」と分類
      ─► (4) セグメント化: 安定区間を状態、遷移を操作として「操作セグメント」を構成
      ─► (5) coalescing: 短い安定を挟んで小変化が連続する区間（タイピング等）を1セグメントに合体
      ─► (6) 代表フレーム選択: 各セグメントの「安定区間に入って最初の」フレーム（静止確認後）
      ─► (7) 証拠付与: before(直前安定区間の代表) / after(代表) / 差分bbox / OCR / 発話
      ─► evidence.json
```

### 各ステップの要点

1. **一様サンプリング**: `-vf fps=N` で一括展開。`-ss` をタイムスタンプごとに起動する現行方式（プロセスspawn×3/フレーム）を廃止し、デコード1パスにする。サンプルは一時ディレクトリに置き、evidence確定後に代表フレームのみストレージへ保存
2. **変化点検出**: dHash（既存 `computeImageDHash` を流用）のハミング距離系列に対し、ヒステリシス付き閾値（遷移開始閾値 > 遷移終了閾値）で安定/遷移を分類。閾値は評価セットで較正し env 化
3. **coalescing 規則**（評価セットで較正）:
   - 安定区間が `COALESCE_MAX_GAP_MS`（目安1000ms）未満、かつ差分bboxが近接（IoU > 0.3 目安）なら同一操作とみなして合体（タイピング、連続スクロール）
   - 合体後のセグメントは「最後の安定状態」を after 代表とする
4. **代表フレーム選択**: 「遷移終了後、ハミング距離が `STABLE_FRAMES` 枚連続で閾値未満になった最初のフレーム」。JPEGサイズ比較は削除
5. **OCR**:
   - 一次: 専用エンジン。`scripts/ocr_server.py`（PaddleOCR、日本語+英語モデル）を子プロセスとして起動しJSON-RPCで複数画像を処理（プロセス起動コストを画像ごとに払わない）
   - フォールバック: 既存 `extractFrameOcr()`（LLM-OCR）。エンジン不在・失敗時に自動切替し、evidence に `ocr_provider` を記録
   - 全画面OCRに加え、差分bbox周辺（パディング付き）のOCR行を `ocr_focus` として別フィールドに保持（執筆フェーズで「操作対象のラベル」として重み付けするため）
6. **ASR割り当て**: 既存 `transcribeVideoSource()` を流用。スニペット選択を「`[t_start - ASR_LEAD_MS, t_end]`（リード目安3000ms）と重なるセグメント」に変更し、隣接セグメントと重複した発話は操作開始時刻に最も近い方へ割り当てる。音声ストリーム無し（既存の検出を流用）なら全セグメント空文字+warning
7. **キャッシュ**: 動画ハッシュ+抽出パラメータをキーに evidence.json 自体をキャッシュ（既存 `pipelineCache` の仕組みを流用）

### evidence.json スキーマ（v1）

```jsonc
{
  "version": 1,
  "project_id": 123,
  "video": { "duration_ms": 312000, "fps_sampled": 3, "sha256": "..." },
  "config": { "dedupe_hamming": 6, "coalesce_max_gap_ms": 1000, "asr_lead_ms": 3000, "ocr_provider": "paddle" },
  "transcript": { "provider": "openai", "segments": [ /* 既存 TranscriptSegment 形式 */ ] },
  "segments": [
    {
      "segment_id": "seg-1",
      "t_start": 0, "t_end": 4200,                  // ms
      "before_frame": { "t": 0, "image_key": "..." },     // 直前の安定状態（先頭セグメントは null）
      "after_frame": { "t": 4200, "image_key": "..." },   // 操作後の安定状態 = 代表フレーム
      "changed_region_bbox": { "x": 0.1, "y": 0.2, "w": 0.3, "h": 0.1 },  // 正規化、null可
      "ocr_lines": ["保存", "キャンセル"],            // after全画面
      "ocr_focus": ["保存"],                          // 差分bbox周辺
      "transcript_snippet": "次に保存ボタンを押します",
      "coalesced_from": 5,                            // 合体した変化点数（タイピング検知の手がかり）
      "warnings": []
    }
  ]
}
```

## タスク分解（PR分割案）

1. **PR-A: セグメンテーションコア** — サンプリング、dHashタイムライン、変化点検出、coalescing、代表選択。単体テストは合成画像列（プログラム生成）で境界を検証。`pnpm eval` のG1（境界F1）でベースライン比較
2. **PR-B: OCRエンジン統合** — PaddleOCRアダプタ + フォールバック + `ocr_focus`。`requirements.txt` 更新、セットアップ手順をREADMEに追記
3. **PR-C: 証拠アセンブリ** — ASRリードウィンドウ、evidence.json 書き出し、キャッシュ、既存 `processVideo` フローへの組み込み（フレームDB保存との互換維持）

## 既存コードとの互換

- `frames` テーブルへの保存は維持する（UIのフレーム一覧タブが参照）。evidence のセグメント代表フレームを従来どおり保存し、`evidence.json` は `projects/<id>/artifacts/evidence.json` に置く
- 旧パス（シーン検出）は evidence 生成失敗時のフォールバックとして1リリースの間残し、安定後に削除（Phase 5）

## 受け入れ基準

- [ ] 評価セット全件で evidence.json が生成され、同一入力で同一出力（決定性）をテストで確認
- [ ] G1（境界F1）がベースライン以上。特にタイピング系ケースで現行比改善
- [ ] タイピング操作が1セグメントに合体されることをテストで確認
- [ ] PaddleOCR 不在の環境で LLM-OCR フォールバックが自動で効く
- [ ] 無音動画で ASR がスキップされ、処理が完走する
- [ ] 10分動画の証拠抽出（OCR込み・LLM-OCR除く）が実用時間内（目安5分以内、環境依存のため計測値を記録）

## リスク

| リスク | 対応 |
|--------|------|
| 2〜4fpsサンプリングでは速い操作（瞬時のクリック2連）を取りこぼす | 閾値とfpsをenv化し評価セットで較正。必要ならfps引き上げ（コストとトレードオフ） |
| coalescing が異なる操作を誤って合体 | bbox近接条件を必須にする。評価セットのフォーム入力ケースで検証 |
| PaddleOCR の日本語UIフォント精度 | 評価セットでG2を測り、悪ければモデル変更 or LLM-OCRへの部分委譲（低confidence行のみ） |
| 1パスデコード+密サンプリングのディスク使用量 | 一時ディレクトリでJPEG品質を落とし（dHash/OCR用途には十分）、処理後即削除 |
