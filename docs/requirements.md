# 要件定義: TutorialGen パイプライン刷新

最終更新: 2026-06-12
ステータス: 確定（実装着手前のベースライン）

## 1. 背景

TutorialGen は画面録画から手順マニュアル（スライド/解説動画）を自動生成するツールである。
現行パイプライン（〜PR #14）は動作するが、生成品質に構造的な天井がある。
2026-06 の専門家レビューで以下の根本課題が特定された。

### 現行アーキテクチャの根本課題

| # | 課題 | 該当箇所 |
|---|------|---------|
| C1 | 1フレーム=1ステップの固定マッピング。意味単位でのステップ統合/分割ができない | `stepGenerator.ts` の生成ループ |
| C2 | 各フレームを文脈ゼロで独立解析。タスク全体・前後ステップをLLMが知らない | `stepGenerator.ts` `analyzeFrame()` |
| C3 | 操作は「フレームの間」で起きるのに、結果画面1枚しかLLMに渡していない。算出済みの差分bbox（`detectChangedRegion`）がプロンプトに未注入 | `stepGenerator.ts:291-301` |
| C4 | ffmpegシーン検出は画面録画のセグメンテーションとして不適切（タイピング・フェード遷移・ページ内アニメーションに弱い） | `videoProcessor.ts` `extractFramesWithFFmpeg()` |
| C5 | 代表フレーム選択が「JPEGファイルサイズ最大」という粗い代理指標。+0.3〜1.0秒後を掴むため次の操作状態を写すリスク | `videoProcessor.ts` `extractBestFrame()` |
| C6 | 発話は操作に先行する（「次に保存を押します」→クリック）のに、transcriptウィンドウがフレーム時刻以降のみ | `asr.ts` `pickTranscriptSnippet()` |
| C7 | 「grounded」がプロンプト上のお願いに留まり、UIラベルのOCR照合などの機械検証がない。confidenceは未較正かつ下流で未消費 | `stepGenerator.ts` |
| C8 | 上流の弱さをテキスト整形ヒューリスティック465行で下流補修している | `slideText.ts` |
| C9 | 生成品質を測る評価基盤が存在しない（テストはユーティリティ単体のみ） | リポジトリ全体 |
| C10 | 動画出力が静止画紙芝居。アーティファクトに `t_start`/`t_end` を持つのに元録画クリップを未使用 | `videoGenerator.ts` |

## 2. 確定済みの前提（2026-06-12 オーナー決定）

| 項目 | 決定 |
|------|------|
| 優先アウトプット | **スライド（PPTX）と解説動画は同等優先**。共通基盤（steps.json品質）を先に固め、スライド→動画の順で改善する |
| 運用形態 | **ローカル/個人ツール**。マルチユーザー本番運用・ストレージ認可（Issue #9）・ジョブキュー基盤は本計画のスコープ外 |
| OCR方式 | **専用OCRエンジン（PaddleOCR等）を導入してよい**。LLM-OCRはフォールバックとして維持 |
| 入力録画 | **ナレーション有無が混在**。音声の有無を自動判定してパスを切り替える設計とする |

## 3. プロダクト目的とゴール

### 目的
操作録画を入れるだけで、**人手修正が最小限で済む**手順マニュアル（スライド/動画）を生成する。

### 測定可能なゴール（評価セットに対して定義）
- **G1: ステップ分割の一致率** — 人手正解とのステップ境界一致（F1）を測定し、改善を継続的に確認できる
- **G2: UIラベル正確性** — 生成文中のUIラベル引用がOCR実測テキストに存在する率
- **G3: 非ステップ混入率** — スクロールのみ・ロード待ち等の非操作ステップの混入率
- **G4: 人手修正コスト** — 評価セットに対し「出荷可能になるまでの編集箇所数」を記録・追跡できる

数値目標は Phase 0 でベースラインを測定してから設定する（現状値が不明なため、先に目標値を置かない）。

## 4. 機能要件

### FR-1 証拠抽出（Evidence Extraction）
- FR-1.1 動画を密サンプリング（2〜4fps目安、設定可能）し、dHashタイムラインから変化点を検出する
- FR-1.2 タイピング等の連続小変化を1つの操作セグメントに合体（coalescing）する
- FR-1.3 各セグメントの代表フレームは「変化後に画面が静止安定した」フレームを選ぶ（JPEGサイズ比較は廃止）
- FR-1.4 各セグメントに before画像 / after画像 / 差分bbox / bbox周辺OCR / 対応発話 を紐付ける
- FR-1.5 OCRは専用エンジンを一次とし、失敗・未導入時はLLM-OCRにフォールバックする
- FR-1.6 ASRは発話が操作に先行する前提のリードウィンドウで各セグメントに割り当てる。音声なし動画では自動でスキップする
- FR-1.7 抽出結果は決定的な中間アーティファクト `evidence.json` として保存し、再実行時はキャッシュを利用する

### FR-2 ステップ執筆（Step Authoring）
- FR-2.1 全セグメントの証拠一式を一括でLLMに渡し、ステップの統合・分割・破棄の裁量を与えて執筆させる（フレームごとの独立解析を廃止）
- FR-2.2 マニュアル全体の概要（タスク名・前提・完了条件）を生成する
- FR-2.3 生成文中のUIラベル引用をOCR実測と機械照合し、不一致は警告+低信頼としてマークする
- FR-2.4 低信頼ステップはUI上でレビュー対象として可視化する
- FR-2.5 出力は `steps.json`（スキーマ拡張版）。既存の編集・並べ替え・再生成UIと互換を保つ

### FR-3 スライド生成
- FR-3.1 既存PPTXレンダラー（default / training プリセット）を `steps.json` v2 から駆動する
- FR-3.2 上流品質の向上に伴い `slideText.ts` の補修ヒューリスティックを段階的に退役させる
- FR-3.3 差分bboxを用いたクロップ/spotlightは維持する

### FR-4 動画生成
- FR-4.1 各ステップの `t_start`/`t_end` で**元録画からクリップを切り出し**、ナレーションを重ねて解説動画を生成する
- FR-4.2 ナレーション音源は「元録画の音声をそのまま使用」「TTS」「無音」をステップ単位で選択できる
- FR-4.3 クリップ切り出しが不可能な場合（フレーム欠損等）は従来の静止画+TTSにフォールバックする

### FR-5 評価
- FR-5.1 録画+人手正解マニュアルからなる評価セット（5〜10本）をリポジトリ管理する（動画実体はサイズ次第で外部置き、参照とハッシュを管理）
- FR-5.2 G1〜G3を自動計算するCLI（`pnpm eval`）を提供する
- FR-5.3 パイプライン変更時にベースラインとの差分レポートを出力できる

## 5. 非機能要件

| ID | 要件 |
|----|------|
| NFR-1 | コスト: LLM呼び出しはフレーム数に対して O(N)×2回（現状）から、OCRのローカル化と執筆の一括化で大幅削減する。キャッシュ（入力ハッシュ+モデル+プロンプトバージョン）は全LLM/ASR/OCR呼び出しで維持 |
| NFR-2 | レイテンシ: 10分・100操作程度の録画を実用時間内（目安: 数分）で処理。独立処理は並列化する |
| NFR-3 | 再現性: 同一入力+同一設定で同一の `evidence.json` が得られる（決定的処理）。LLM出力はキャッシュで再現性を担保 |
| NFR-4 | 堅牢性: 個々のセグメント解析失敗が全体を落とさない（現行のフォールバックステップ方式を踏襲）。TTS等のサイレントフォールバック発動はUIに警告として可視化する |
| NFR-5 | メモリ: 動画をメモリに全載せしない（アップロード・処理ともディスクストリーミング） |
| NFR-6 | セットアップ: ローカルツールとして `pnpm install` + Python依存（requirements.txt）+ ffmpeg で完結。専用OCRエンジンが無い環境でもLLM-OCRフォールバックで動作する |

## 6. 非目標（明示的にやらないこと）

- マルチユーザー本番運用のための認可強化（Issue #9）— 据え置き。設計上の互換は壊さない
- ジョブキュー/ワーカー基盤、水平スケール
- マルチパート・再開可能アップロード
- リアルタイム処理（録画しながらの逐次生成）
- クリック座標の自動検出を**必須要件**にすること（カーソル検出は精度向上のオプション研究項目とする）

## 7. 既存資産の扱い

| 資産 | 扱い |
|------|------|
| プロバイダー抽象（LLM/TTS/ASR: `server/_core/`） | 維持・流用 |
| パイプラインキャッシュ（`pipelineCache.ts`） | 維持・流用 |
| steps.json アーティファクト中心の設計（`stepsArtifact.ts`） | 維持。スキーマをv2に拡張 |
| スライドレンダラー（`slideGenerator.ts` + プリセット） | 維持。入力をv2に差し替え |
| 認証/ストレージ/環境変数基盤 | 維持 |
| フレーム抽出〜ステップ生成の中核（`videoProcessor.ts` のシーン検出、`stepGenerator.ts` の生成ループ） | **置き換え**（Phase 1-2） |
| `slideText.ts` ヒューリスティック群 | 段階的退役（Phase 3） |
| `scripts/extract_frames.py` | 削除（未参照の死にコード。現行はffmpeg直接実行） |
| DBの `steps` テーブルと steps.json の二重管理 | steps.json を単一の真実とし、DBは表示用インデックスに縮退（Phase 5） |

## 8. 成功判定

本計画の完了条件:
1. 評価セットでG1〜G3がベースライン比で改善している（数値目標はPhase 0完了時に設定）
2. スライド・動画とも `steps.json` v2 から生成され、既存UIの編集フローが機能する
3. ナレーション有無いずれの録画でもエンドツーエンドで生成が完了する
4. 上記が `pnpm eval` で再現可能に検証できる
